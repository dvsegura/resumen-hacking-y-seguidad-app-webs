Introducción al pentest web ^

1.1 ETAPAS DEL PENTEST

Clasificación dependiendo del nivel de información de partida a la hora de realizar el test de penetración

  • Caja Negra(no se dispone de información)
  • Caja Blanca(el penester dispone del total de información sobre el sistema a auditar)
  • Caja Gris(mezcla de las dos anteriores)

Metodologías

  • NIST(National Institute of Standards and Technology)
  • OWASP(Open Web Application Security Project)
  • OSSTMM(Open Source Security Testing Methodology Manual)

Etapas

Reconocimiento (Primera)

Se obtendrá toda la información disponible en medios de acceso público sobre el objetivo. Se podrán emplear técnicas de reconocimiento pasivo y activo.

Pasiva
  • Hacking con buscadores
  • Uso de herramientas de gestión de red(nslookup, dig)
  • Herramientas de búsqueda de metadatos o indagación en redes sociales
Activa
  • Atacante interactua con la víctima(ingeniería social)
Fase Descripción Objetivos Herramientas

Explotación Se emplearán exploits contra servicios y aplicaciones vulnerables, fuzzing para detectar posibles errores no controlados. Elevación de privilegios e intentar mantener ese acceso no autorizado mediante la instalación de backdoors y rootkits Metasploit, Canvas
Documentación Documentar información obtenida en las etapas anteriores ¡, describir paso a paso los ataques realizados junto a los resultados producidos. Cliente conozca las vulnerabilidades de su aplicación Finalizar con consejos para solucionar las vulnerabilidades descubiertas.

1.2 INTRODUCCION PROTOCOLO HTTP(HyperText Transfer Protocol)

Es un protocolo de la capa de Aplicacion de Modelo OSI. Este protocolo permite realizar peticiones y respuestas en una arquitectura cliente-servidor. Existen los campos de cabecera de peticion y de respuesta HTTP.

Códigos de respuesta HTTP:

Código Mensaje
1xx Mensaje informativo.
2xx Petición recibida y procesada correctamente.
3xx Indica que hay que hacer una redirección.
4xx Se ha producido un error en el cliente.
5xx Se ha producido un error en el servidor.

1.4 FIGENRPRINTING WE SERVERS

El término fingerprinting hace referencia a la huella distintiva que permite identificar un sistema. En este caso, estaremos profundizando en el HTTP Fingerprinting

1.4.1 Tipo y versión del servidor web

Etiquetas Server y Vía de la cabecera HTTP. Este método de fingerprinting consiste en impeccionar la cabecera HTTP devuelta por el servidor en busca de la etiqueta Server. De forma genérica esta técnica de conoce como Banner Grabing

el campo Vía de la cabecera es utilizado por proxiesy gateways para indicar los protocolos entre el User Agent y el Server.

Identificación a través de páginas de error.La idea consiste en eviar peticiones mal formadas para que el servidor responda con una página de error(error 404) donde se pueden revelar informacion sobre el SW empleado en el servidor. Por eso es tan importante establecer páginas de error personalizadas.

Análisis del comportamiento de la respuesta HTTP. Consiste en realizar diversos tipos de peticiones y estudiar las respuestas devueltas por distintos modelos de servidores. Ejemplos a continuación

  • Distintos campos en la cabecera
  • Orden de los campos de la cabecera
  • Lista de los métodos permitidos
  • Respuestas antes peticiones DELETE

Tecnologías empleadas en el servidor web

Otro aspecto importante para poder orientar adecuandamente los ataques que realizará el auditor es el de determinar las tecnologías con las que trabaja el servidor. A partir de esta información el pentester podrá buscar vulnerabilidades propias de esos lenguajes y seleccionar la sintaxis de sus inyecciones de código malicioso.

Formas de determinar las tecnologías con que trabaja el Servidor web:

  • Cabecera X-Powered-By
  • Extensiones de los scripts
  • Contenido de las cookies

CROSS SITE SCRIPTING ^

Los ataques de Cross-Site Scripting o XSS constituyen un tipo de inyección de código,generalmente JavaScript(aunque puede ser aprovechado por otros lenguajes como HTML, etc.) que tiene como objetivo no atacar la maquina que presta el servicio web, sino al resto de usuarios que acceden a dichos servidores web, utilizando como pasarela para ello una aplicació vulnerable que este ejecutandose en el servidor implicado.

El pricipal motivo de la existencia de esta vulnerabilidad es que la aplicacion web no filtra correctamente una cadena de texto recibida, en la que un posible atacante ha introducido intencionalmente instrucciones interpretables mediamte un lenguaje de programación dado.

Uno de los valores que aporta esta vulnerabilidad a los atacantes es su gran flexibilidad, que dependiendo de la aplicacion y el contexto en que se ecuentre permitiran desde el robo de variables de sesión(suplantación de usuario) hasta redireccion a sitios web fraudulentos(ataques de phishing bancario)

3.2 XSS REFLEJADO

El Cross-Site Scripting Reflejado es aquel en el cual al servidor le llega un fragmeto de código interpretable, generalmente HTML o JavaScript y lo reenvia introducido directamente en el código fuente de una página web vulnerable hacia la parte cliente de la aplicación, que suele ser le navegador web del cliente. El codigo sera interpretado por el navegador web del usuario y los efectos implicados se manifestarán en el mismo. El código inyectado no se almacena en ningun lado, siendo enviado directamente a la aplicacion web por algunos de los posibles campos recibidos.

Ejemplo. Campo de Búsqueda de contenido. Si se introduce en el campo de búsqueda: 

           <h1>codigo_inyectado</h1>

Luego se muestra en el cuerpo de la pagina el mensaje: Los resultados de buscar codigo_inyectado sin los siguientes:

Si en este mensaje la cadena codigo_inyectado se mostarse como un encabezado de primer nivel, significaria que la aplicación a compuesto esta pagina de resultados concatenando directamente la cadena introducida para buscar sin tratarla. Con esto se habria conseguido introducir codigo malicioso en la aplicacion que ha afectado directamente al cliente, sin que dicho código haya tenido que ser almacenado en al aplicación.

Dado que el contenido inyectado no se guarda en el servidor y va introducido en la misma peticion HTTP, por ejemplo:

http://www.miweb.com/buscar.php?busqueda=<h1>codigo_inyectado</h1>

Para consegir que una tercera persona, usuario víctima del ataque, reciba el resultado de interpretar este código en su navegador, deberia ser el quien realizase la petición HTTP. Para ello, en este tipo de ataque suele emplearse el envio de URL malisiosa por email, publicación de enlaces o foros de internet, etc.

Ver ejemplos de la plataforma de pruebas Damn Vulnerable Web Application(DVWA)-Nivel Low

Como puede comprobarse, toda variable o parámetro que reciba la aplicacion debe ser correctamente filtrado, evitando que contenga valores que puedan causar comportamiento inesperado.

Hacking con HTML5

Presentando HTML5

HTML5 es el resultado del trabajo conjunto del World Wide Web Consortium(W3C) que se encarga principalmente del XHTML2.0 y del Web Hypertext Application Technology Working Group(WHATWG) que se centra en el tratamiento de aplicaciones y formularios web.

Objetivos:

  • Multiplataforma e independiente del dispositivo utilizado.
  • Se debe proporcionar todo tipo de contenidos audiovisuales, empleando las etiquetas prevista para cada caso. Se busca que no sea necesario instalar plugins de terceros.
  • No debe limitarse a proporcionar información que solo pueda ser reproducida on-line sino que tiene que ser funcional incluso offline a través de una caché.
  • Apliación de las posibilidades de estilos CSS, dando lugar al CSS3
  • HTML5 debe ser compatible con HTML4

Algunas etiquetas nuevas del HTML5:(article,header,nav,footer,aside,audio,video,canvas)

Otras caracteríticas:

  • Web Sockets(establece un canal de comunicación permanente por el que intercamiar contenido)
  • Web Workers(permiten ejecutar scripts en forma de threads que correran en segudo plano concretamente, en un contexto global distinto al del objeto window actual)

Socket: TCP/IP puede entenderce como uno de los extremos de una comunicación bidimencional entre dos procesos. Cada socket estara identificado pr la direccion IP a la que pertenece.

Nota: HTML5 puede habilitar una cache conocida como appcache para trabajar offline. Forma de activarla mediante el atributo manifest de la etiqueta html

6.3 Técnicas de evasión de WAF basado en HPP y HPF

Un ataque HPP se puede utilizar para hacer un bypass del WAF o conseguir generar comportamientos inestabls en la aplicacion web en caso que no haya un WAF. Este ataque consiste en inyectar varios parametros repetidos y en uno de ellos introducir codigo para vulnerar la aplicacion web. En el otro parámetro un dato normal

ver Práctica Ataque HPP

  • Test 1 - Introducir 2 veces el parámetro ID con distintos valores númericos(index.jsp?id=1&id=2). Resultado(el script ha escogido el último campo(php))
  • Test 2 - Introducir una "SQL Injection" en el ultimo parametro repetido(index.jsp?id=2&id=' or 1=1 -'). Resultado(Ataque exitoso - debido a que el WAF solo analizó el primer parámetro que era correcto)
  • Test 3 - Introducir igual que el Test 2 pero invirtiendo el orden de los parámetros. Resultado(Ataque Fallido - debido a que en este caso el primer parametro no cumplia con la expresion regular)

Peticiones http parameter fragmentation esta técnica consiste en ir introduciendo una vulnerabilidad en la aplicación web po trozos, es decir, dividir una inyección SQL en diferentes partes e introducir cada parte en cada uno de los parámetros que recibe la aplicación.

Ejemplo de un buscador con 3 campos de búsqueda. Se podrá inyectar de esta manera:

  • El campo 1 contiene el fragmento de inyección: ' or 1/'
  • El campo 2 contiene el fragmento: */ =0 union /*
  • El campo 3 contiene: */ select user(),database(),version()-/*

Nota: para probar el funcionamiento de buscarlibros.php con el WAF, hay que realizar un cambio en el formulario de la página, sustituyendo el valor del action por la direccion donde se encuentra el WAF. Ej.

form action="http://localhost:8080/waf/index.jsp" method="get"

En el ataque se ha mostrado como dividiendo la inyección en partes es posible llegar a saltarse la seguridad que aplicaba al WAF. Se podría crear otro tipo de filtros para realizar búsquedas por palabras o símbolos en vez de por una vulnerabilidad en concreto. Es decir, un filtro que al encontrar la palabra(select, union, etc) bloqueara la petición. El problema que puede surgir con estos filtros tan exhaustivos es que pueden llegar a restringir tanto que no permitirá funcionar al cien porcieto la aplicacion web. SOLUCION (Listas Poderadas): crear varios filtros y cada uno de ellos devovera un número. Ese numero se acumular y se sumara con el resto de números devueltos por cada filtro. Si la suma supera un cantidad establecida, se considerara un ataque.

Se han visto las técnicas de evasion WAF HPP Y HFP pero tambien existen otras técnicas:

  • Escribir las inyección SQL en mayúsculas y minúsculas
  • bytes nulos(El byte null equivale a la cadena "%00") y representa la terminación de una cadena o caracter. Por tanto lo que estuviera detrás del byte nulo quedaría anulado por el lenguaje de programación. Ejemplo de inyección: "%00' or 1=1#"
  • Hay WAF que leen de izquierda a derecha. En ese caso es posible realizar un ataque tipo XSS. El WAF puede tener un filtro que busque la palabra script y al encontrarla la sustituya por vacio. Para poder evadir este filtro se puede anidará el ataque de la siguiente manera: <scrscriptipt>alert(0)</scrscriptipt>. Al el WAF eliminar "script" se juntaría la cadena y daría lugar al ataque XSS bien formado.

CONCLUSIONES

  • Si no se realiza una correcta configuración y mantenimiento del WAF, no valdrá la pena tenerlo.
  • También es posible evadir los mecanismos de seguridad de un WAF
  • Es importante no solo tener esa capa de protección que aporta el WAF, sino además tener filtros en las páginas webs que sirvan de protección en caso que un ataque consiga traspasar la seguridad de la primera barrera(WAF).
